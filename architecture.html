Architecture

How GlowFlow turns GlowScript into automation.

GlowFlow is intentionally built around a small language stack, not a pile of ad hoc scripts. The architecture is simple enough to explain, but strong enough to support readable syntax, semantic checks, direct execution, and a path to native binaries.

The current release should be read as GlowFlow v1.0.1 and GlowScript v1.0.1: a public release line with a real language boundary, a proven execution surface, HTTPS webhook transport, and language-level error recovery, while still leaving room for future hardening.

Performance should be read the same way. The target is not magic zero-latency automation; it is keeping language overhead small enough that servers spend their time on AI calls, APIs, files, and tools instead of on the GlowFlow runtime itself.

1. Lexer

The lexer reads GlowScript line by line and emits indentation-aware tokens. This is what makes the language feel plain-English without becoming ambiguous.

  • handles two-space indentation blocks
  • recognizes words, strings, numbers, lists, objects, and operators
  • produces friendly early errors for bad indentation and broken strings

2. Parser

The parser maps readable statements like `set`, `task`, `when`, `extract`, and `pipe` into a compact abstract syntax tree.

  • keeps the grammar small and explainable
  • supports indentation blocks instead of punctuation-heavy syntax
  • preserves a clean shape for formatting and inspection

3. Semantic Analyzer

The semantic pass is where GlowScript stops being just parseable text and starts behaving like a language with rules.

  • catches unknown names before runtime
  • checks workflow shapes like `repeat`, `for each`, and `run task`
  • enforces object access and basic numeric comparison rules

4. Interpreter Runtime

The interpreter makes GlowScript immediately useful. `glow run` executes the language directly with files, JSON, CSV, AI calls, tools, scheduling behavior, memory, and webhook handling.

  • fast path for MVP execution
  • most complete runtime today
  • used as the reference behavior for test parity

5. AI And Tool Layer

GlowScript treats AI and automation tools as language concepts, not bolt-ons. Structured forms like `extract`, `classify`, and `decide` flow through a shared runtime contract.

  • supports fallback, mock, command, OpenAI, and Ollama paths
  • validates structured responses instead of trusting raw text
  • supports MCP-style filesystem tools and user-defined tools

6. Build Path

GlowFlow has a deployment story beyond the interpreter. The build path is designed to preserve behavior while moving toward native execution.

  • analyzes the same language pipeline before build output
  • supports transpilation and a native runner path
  • native parity is checked against interpreter output in tests

Why This Architecture Matters

The important part is not that GlowScript has a lexer and parser. The important part is that the language boundary is real. GlowFlow is not “just some prompts” and it is not “just YAML with better words.” It already has the structure of a serious language: 

GlowScript source
  -> lexer
  -> parser
  -> AST
  -> semantic analyzer
  -> interpreter runtime or native build path

That architecture is what makes GlowFlow presentable as a new category: a scripting language built specifically for AI automation.

For this MVP, the practical goal is simple: startup should feel instant, parsing and checks should be negligible, and webhook handlers should stay fast unless external systems are the bottleneck.
