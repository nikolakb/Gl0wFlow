Handbook

The serious handbook for reading, writing, and reasoning about GlowScript.

This page is the operating guide for GlowFlow v1.0.1. It explains how GlowScript is meant to be read, how the execution model works, how data moves through a script, and where the language is intentionally strict.

The goal is not to describe every token. The goal is to explain the logic that makes GlowScript behave like a real scripting language for AI automation instead of a loose pile of prompts, YAML, and shell glue.

Readable by design Programs should read like instructions, not framework plumbing.
Automation-native Tasks, webhooks, AI, tools, files, and schedules are first-class.
Language rules Lexer, parser, AST, semantic checks, runtime, and native build all exist.
Proven surface GlowFlow currently stands on one mother end-to-end proof, one targeted regression test, and live runtime proof.
Mental Model

Think in steps and effects.

A GlowScript file is a sequence of named steps. Some steps create values. Some steps make decisions. Some steps trigger outside-world effects. That is the center of the model.

  • `set` creates state.
  • `if`, `repeat`, and `for each` route control flow.
  • `task`, `function`, and `tool` package reusable logic.
  • `ask ai`, `extract`, `classify`, `summarize`, `rewrite`, and `decide` perform reasoning work.
  • `when webhook` and `every` expose live automation entry points.
Backbone

The language stays intentionally small.

GlowScript is serious because it stays constrained. The backbone words are stable and readable: `set`, `say`, `if`, `repeat`, `for each`, `task`, `when`, `every`, `ask`, `use`, and `tool`. 

set lead to {name: "Ana", score: 92}

if lead.score > 80
  say "Priority lead"
else
  say "Standard lead"
Execution Model

One language, two execution paths.

GlowFlow sends source through a language pipeline: lexing, parsing, AST construction, semantic analysis, then either direct interpreter execution or a native build path. The interpreter is the most complete runtime today. Native output is still tested against the same language behavior. 

GlowScript source
  -> lexer
  -> parser
  -> AST
  -> semantic analyzer
  -> interpreter runtime or native build output
Data Flow

`result` is the working register.

Many operations write their output into `result`. That keeps short workflows compact and makes pipeline-style flows possible without introducing a temporary name for every intermediate step. 

read file "report.txt"
pipe summarize
pipe save "summary.txt"

Use explicit variables when a value matters later. Use `result` when the value is just moving through the current flow.

Core Values

Values are simple and practical.

GlowScript works with strings, numbers, booleans, lists, objects, and structured response objects. This is deliberate. Automation languages spend their time moving data between systems, not building elaborate object hierarchies.

  • Files: `read file`, `save`, `save json`, `save csv`.
  • Data: `load json`, `load csv`, object and list literals.
  • HTTP: `get`, `post`, `put`, `delete` return structured response objects.
  • System state: `env`, `now`, `length`, and memory-like `remember` support.
Automation Logic

Tasks and events are first-class.

A serious automation language should not hide workflows behind a host framework. GlowScript keeps the automation surface visible in source. 

task sync_sales
  get "http://127.0.0.1:8080/sales"
  save result to "sales.json"

every 1 hour
  run task sync_sales

when webhook "/signup"
  read json body
  save json body to "signup-{now}.json"
  reply "Signup received"
AI Logic

AI is a structured language feature.

GlowScript does not treat AI as an afterthought library. The language has direct forms for asking, extracting, classifying, summarizing, rewriting, and deciding. That lets scripts express intent instead of repetitive provider plumbing. 

extract from message using ai:
  provider "ollama"
  model "qwen3-coder:480b-cloud"
  system "Extract only verified lead details"
  fields:
    name
    email
    company
Tool Orchestration

Tools are capability boundaries.

Use tools when a workflow needs to cross into another system or when logic should be reusable from outside the current script.

  • `use mcp filesystem` activates filesystem-style tool space.
  • `call tool "filesystem.read_file"` bridges into adapters.
  • `tool` plus `export mcp tool` exposes GlowScript-defined capabilities outward.
Safety Model

Failures should be directed and visible.

A serious workflow language should fail clearly. GlowScript already enforces semantic guardrails around undefined names, invalid property access, incorrect loop shapes, and misuse of comparisons. Runtime rules also stay explicit, such as POST-only webhook execution and visible approval checkpoints.

  • Prefer failing before the side effect happens.
  • Prefer explicit approvals for human checkpoints.
  • Prefer readable diagnostics over cryptic parser noise.
Performance

Fast means low language overhead.

GlowFlow is designed so runtime overhead stays small and predictable. In real server use, most latency will come from AI providers, APIs, disk I/O, and external tools. The language should not be the slow part.

  • Small scripts should start near-instantly.
  • Parse plus semantic checks should stay comfortably sub-second.
  • Simple webhook handlers without outside calls should return quickly.
  • The heavy time budget belongs to the external work being orchestrated.
Authoring Rules

How to keep scripts maintainable.

  1. Name variables after business meaning, not transport details.
  2. Use tasks, functions, and tools to isolate repeated logic.
  3. Prefer structured AI forms when output shape matters.
  4. Persist important workflow artifacts explicitly with JSON, CSV, or files.
  5. Use `approve` where the workflow crosses a human risk boundary.
  6. Keep each file centered on one workflow or one capability boundary.
Release Boundaries

What v1.0.1 proves.

GlowFlow v1.0.1 establishes GlowScript as a real, executable language with a functioning runtime, a native build path, provider-backed AI proof, HTTPS webhook transport, language-level error recovery, and serious test coverage. It does not claim that every adapter, integration, or operations edge case is already finished.

  • It proves GlowScript can be executed, tested, and built as a language.
  • It proves coverage across the CLI, parser, interpreter, runtime, and native parity path.
  • It is honest about the remaining work around the broader ecosystem.
Short Definition

The serious one-line description.

GlowScript is the scripting language for AI automation. GlowFlow is the product, runtime, and documentation surface that makes that language usable in real systems.
