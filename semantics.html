Semantics

The detailed semantics and capabilities of GlowScript v1.0.1.

This page explains the language as a language: what statements mean, how values flow, what the runtime guarantees, which operations are strict, and where the current implementation stops.

Core Model

Statement-first, automation-first.

  • GlowScript is built around named workflow statements instead of punctuation-heavy syntax.
  • Programs are indentation-based and read as ordered steps.
  • The language is optimized for automation readability, not abstraction density.
Values

Simple values, explicit structure.

  • Primitive values: strings, numbers, booleans, `null`.
  • Structured values: lists and objects.
  • HTTP responses are object-shaped values with `status`, `status_text`, `headers`, `body`, and `text`.
  • AI outputs commonly land in `result` unless the statement writes elsewhere.
Control

Readable control flow.

  • `if` / `else` branch on truthy values.
  • `repeat` loops numerically.
  • `for each` requires a list.
  • `retry` reruns a block until success or exhaustion.
  • `parallel` is grouped intent in v1.0.1, not true concurrent scheduling.
Workflow Units

Tasks, functions, and tools.

  • `task` defines reusable workflow steps.
  • `function` defines reusable logic with named parameters.
  • `tool` defines reusable tool-shaped capabilities.
  • `call function` and `call tool` pass named argument objects.
Files And Data

Files, JSON, CSV, and HTTP are native.

  • `read file`, `save`, `load json`, `save json`, `load csv`, `save csv` are built-in language forms.
  • `get`, `post`, `put`, and `delete` are expressions and can be assigned or recovered.
  • Paths resolve relative to the script working directory.
Webhooks

GlowScript is a server language too.

  • `when webhook "/path"` defines a request entry point.
  • `read json body` populates `body`.
  • `reply expr` returns the response body.
  • `glow serve` now supports both HTTP and HTTPS transport.
AI

Structured AI is language surface, not helper code.

  • `ask ai`, `extract`, `classify`, `summarize`, `rewrite`, and `decide` are first-class.
  • AI config can include `provider`, `model`, `system`, and `retries`.
  • The current providers are `mock`, `command`, `openai`, and `ollama`.
Errors

v1.0.1 adds first-class recovery.

  • `throw expr` raises a runtime error using the rendered value text.
  • `try` / `catch error` catches runtime failures and binds the message to the catch variable.
  • `expr recover fallback` returns the fallback only if the wrapped expression fails.
  • `recover` is most useful around files, HTTP, and tool calls.
try
  set data to read file "missing.json"
catch error
  set data to "{status: \"fallback\"}"

set status to get "http://127.0.0.1:9/health" recover {status: "offline"}
Semantics

What gets checked before runtime.

  • Undefined values are rejected early.
  • Missing tasks, tools, and functions are rejected early.
  • Numeric comparisons expect number-compatible values.
  • Property access expects object-like values.
  • `remember` only accepts already-created values.
Execution

Interpreter first, native runner second.

  • `glow run` executes through the interpreter.
  • `glow build` produces a native runner path that preserves the current supported semantics.
  • The language is designed to be deployable, not only interactive.
Current Capabilities

What GlowScript can do now.

  • Readable automation with variables, branching, loops, tasks, functions, tools, and imports.
  • HTTP and HTTPS webhook handling.
  • Files, JSON, CSV, HTTP, AI, approvals, and memory.
  • MCP-style filesystem tools and exported user-defined tools.
  • Check, inspect, run, build, format, and REPL entry points.
Boundaries

What v1.0.1 still does not claim.

  • No daemon scheduler yet for true recurring background execution.
  • No fully broad MCP ecosystem yet.
  • No claim of full production-hardening across every adapter and operational edge case.
  • No attempt to replace a general-purpose language ecosystem.
