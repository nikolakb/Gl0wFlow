Speed

GlowFlow runtime overhead is small enough to stay out of the way.

These numbers were measured locally against the actual GlowFlow CLI and native runner. The goal is not claiming impossible zero-latency automation. The goal is showing that GlowScript itself is not the bottleneck.

3.13 ms Mean `glow check` time on `examples/hello.glow`.
3.00 ms Mean `glow run` time on `examples/hello.glow`.
2.99 ms Mean native runner startup time for the built `hello` binary.
14.15 ms Mean localhost webhook roundtrip for a trivial POST handler.

Interpretation

What these numbers mean

  • GlowFlow startup and local script execution are effectively instant to a human user.
  • The native runner path is in the same low-single-digit millisecond range.
  • The trivial webhook path is fast enough that GlowFlow itself is not the main cost center.
  • In real systems, total latency will mostly come from AI providers, APIs, files, and external tools.

That is the useful speed claim for this product: GlowFlow keeps language overhead small and predictable, so the work being orchestrated dominates the time budget instead of the language runtime itself.

Practical Statement

The short version

GlowFlow is already fast enough for server-side automation. The language is not the bottleneck. AI calls, HTTP services, file I/O, and tool integrations are the bottleneck.
