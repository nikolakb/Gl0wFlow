Language Proof

What the tests prove about GlowScript inside GlowFlow.

This is not just a repo health number. The test suite is the current proof that GlowScript behaves like a real language: it accepts the intended syntax, rejects invalid programs with beginner-friendly diagnostics, evaluates workflow logic, preserves formatting, validates AI contracts, resolves imports, and keeps native builds aligned with interpreter behavior. For an AI automation language this is the critical threshold: the idea has crossed into a proven executable system.

Syntax And Structure

The mother test proves the language surface in one place.

  • The retained end-to-end scenario starts with real GlowScript source, not hand-built runtime calls.
  • `check` and `inspect` both run against that same source before execution begins.
  • This keeps the proof centered on the language itself rather than on isolated helper behavior.
Regression Safety

One low-level parser edge is still protected directly.

  • The internal regression test protects webhook body parsing around request-line and `Content-Length` handling.
  • That bug was found during heavier runtime stress and kept as a direct unit-level guard.
  • So the suite still preserves one surgical test where a narrow failure mode matters.
Execution

The end-to-end flow exercises real workflow behavior.

  • The mother flow runs imports, functions, tools, MCP filesystem access, `retry`, `parallel`, `every`, `approve`, `wait`, JSON, CSV, and file persistence.
  • It also verifies that the expected workflow artifacts are actually written.
  • This proves GlowScript is not only parseable; it is runnable across a serious mixed workflow.
AI Layer

AI and tool orchestration are still part of the proof story.

  • The mother flow covers `extract`, `classify`, `rewrite`, and `decide` inside one executable scenario.
  • Separate live stress work also proved the local Ollama path for structured AI operations.
  • The public proof remains honest: the runtime is real even if the broader adapter ecosystem is still growing.
Build Parity

Native output is compared directly against interpreter output.

  • The mother test builds a native runner for the same script it interprets.
  • The resulting stdout is compared directly for parity.
  • This is the strongest current automated proof that GlowScript behaves like one language across both execution paths.
